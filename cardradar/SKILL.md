# CardRadar — 個人信用卡回饋查詢

這是一個個人自用的信用卡哨兵。
目標：使用者丟一個消費情境（通路＋金額），就回他「該刷哪張卡、實際拿多少」。

**啟動：`/card ...`**
- `/card <消費情境>` → 查最優卡。例：`/card 全聯怎麼付`、`/card 高鐵票6000刷哪張`。
- `/card 新增 <卡名>` → 查官方資料、寫入該卡回饋列（有疑惑當場釐清後寫入）。
- `/card 更新 <卡名/通路>` → 重查並更新既有列（有疑惑當場釐清）。
- `/card 失效 <列>` → 活動結束標已失效。
- `/card 摘要`（或 `/card 報告`） → 依你記住的消費習慣，產出信用卡摘要報告與推薦（見「消費習慣摘要報告與推薦」）。

查詢類照「查詢流程」走；寫入類（新增/更新/失效）照下方「寫入操作」走；摘要類照「消費習慣摘要報告與推薦」走。
**每次更新回饋資訊（新增/更新/失效）寫入並回讀確認後，率先自動對受影響的消費習慣做一次摘要報告與推薦。**

## 設定來源：先讀本機設定檔（重要）

本 skill 不寫死任何 ID 或個人資料。啟動時先讀**同目錄**的檔（前兩個必備，第三個選填）：

- `config.local.json` — Notion 基礎設施：
  - `ruleDataSourceId`：規則表 data source id（給 `notion-search` 的 `collection://` 用）。
  - `ruleDbUrl`：規則表 DB 連結。
  - `projectPageId`：CardRadar 專案頁（選填）。
  - `rowIndex`：逐列 page ID 索引（選填 fallback，結構＝`{ 卡名: { 通路標籤: page_id } }`）。
- `profile.local.json` — 持卡人個人化資料：
  - `persona`：稱呼／身分（影響語氣）。
  - `bankTier`：各行會員等級、戶況、預設成立的加碼條件（如電帳＋自扣）。
  - `cards`：實際持有的卡清單。
  - `paymentApps`：實際使用的行動支付（**只認這些**，見鐵則 2）。
  - `paymentCardPairing`：每個行支配哪張卡最優。
  - `pointMappings`：各卡回饋點數形式（選填）。
- `habits.local.json`（**選填**）— 消費習慣記憶檔：`常用通路`／`常用支付`／`消費面向`（各含 `count` 權重與 `lastSeen`）、選填 `月度概況`／`偏好`。詳見「消費習慣記憶」一節。**缺檔不影響查詢**——第一次要記習慣時才依 `habits.example.json` 建立。

**若 `config.local.json` 或 `profile.local.json` 不存在**，提示使用者：「請複製 `config.example.json`／`profile.example.json` 為對應的 `.local.json` 並填入你的 Notion ID 與持卡資料」，並停止後續讀寫動作。`habits.local.json` 缺檔則不擋——照常查詢，並在該記習慣時建立它。

## 第零鐵則：資料在 Notion，不在你腦袋

**每次查詢都必須先讀取即時 Notion 資料，絕不能憑記憶或本檔內的範例數字回答。**
回饋細節（率、上限、到期日、狀態）會變，唯一可信來源是 `config.local.json` 指向的那個規則表 DB。

**讀取方式（重要，依使用者方案調整）：**
整表 SQL 與 view 查詢在部分帳號被擋（`notion-query-data-sources` 需 Business＋Notion AI，會回 402/validation error）。
所以**不要預設用 SQL**。改用這個順序：

1. `notion-search`，帶 `data_source_url: "collection://<config.ruleDataSourceId>"`、
   query 放情境關鍵字（例如「海外」「全聯 超商」「高鐵 交通」），找出相關列。
2. 對找到的列用 `notion-fetch`（帶 page ID）逐列讀出 properties（率、上限、排除、到期日、狀態）。
3. 若 search 不靈，且 `config.rowIndex` 有對應卡列，直接用其中的 page ID 逐列 fetch。

`notion-fetch` 讀單一 page（含資料源頁）一定可用，不受方案限制。
若 Notion 工具未載入，先 `tool_search("notion fetch search")` 載入再用。
若真的讀不到，**明說讀不到、不要編數字**。

## 三條查詢鐵則

### 1. 算「實拿」，不是比率
回饋有上限，光排回饋率會騙人。每張候選卡的實拿：

```
實拿 = min(消費金額 × 回饋率, 每月回饋上限)
```

（上限為空＝無上限，則實拿 = 金額 × 率。）
多張比較時**一律用實拿金額排名**，不是用率排名。

**範例邏輯**（數字僅示意，實際以 Notion 為準）：海外刷 2 萬，
A 卡 6%／上限 800 → 實拿 min(20000×6%, 800)=800；
B 卡 4.5%／上限 400 → 實拿 min(20000×4.5%, 400)=400。A 贏。
但若只刷 5,000：A=300（未到上限）、B=225 → A 仍贏；
刷越少越接近純比率比較，刷越多越要看上限。

若使用者**沒給金額**，就同時給：比率、上限、以及「約刷多少會封頂」，
並提醒「實際最優取決於你這筆金額」。

### 2. 只認使用者在用的行動支付
使用者實際使用的行動支付列在 `profile.paymentApps`。
**不在這份清單裡的行動支付**（例如街口、橘子、悠遊付、歐付寶、Apple/Google/Samsung Pay、Hami Pay 等他沒在用的），
**那些支付綁定才有的加碼一律不計入比較與排名**，不可拿來灌高帳面回饋。

哪個 App 配哪張卡最優，依 `profile.paymentCardPairing`。

### 3. 誠實標示資料狀態
- 狀態「已失效」**不列入**比較。
- 比對「活動到期日」與今天（先用 `user_time_v0` 取得今天）：
  - 已過期 → 該列加碼可能失效，提醒重查、不要當有效回饋算。
  - 7 天內到期 → 標「即將到期」，提醒可能要重查下季條件。
- 更改或新增信用卡資訊時若對任何細節有疑義，**務必主動提出並當場向使用者釐清再寫**，不先寫個草稿放著（見「寫入操作」）。

## 通路特例（台灣市場已知現實，免每次現查）
這些是 Notion 表外的真實世界規則，會直接改變答案，視為已知（以下為台灣市場通則，與個人無關）：

- **全聯／大全聯**：
  - 全聯收的行動支付有限，部分支付不收（以當期門市公告為準）；只收得到的支付才談得上回饋。
  - 「全支付筆筆 5%」這類普惠回饋通常**明確排除全聯場域** → 在全聯用該支付綁卡拿不到那 5%，只剩底層卡回饋。
  - **2026/1/1 起，永豐、富邦、中信等多家銀行已把「全聯體系（門市／PX Pay／全支付）」排除在一般消費回饋外** → 這幾家的卡**直刷全聯 ≈ 0 回饋**。
  - 結論：全聯不是賺回饋的地方。實際可賺的多半只有全聯不定期的「卡友日」（限量、需看當期公告）。平常就用收得到的支付付，圖方便與累點。
- 其他通路若日後發現類似「表面有回饋、實際被排除或支付不被收」的情況，比照辦理：誠實告知拿不到，不要照帳面率報。

## 使用者已知前提（仍以 Notion 為準，這裡只是背景）
這些前提來自 `profile.bankTier`（會員等級、戶況、預設成立的加碼條件等），讀檔後套用：
- 各行會員等級／戶況決定基礎率與上限（如大戶級率/上限、幣倍卡 Level 對應上限）。
- 多卡若已設電子帳單＋自動扣繳 → 需此條件的加碼，預設視為成立。
- 卡片版本（現金版／點數版、新戶／舊戶）影響適用加碼，以 `profile` 記載為準。
- 點數形式對應見 `profile.pointMappings`；折現難易可順帶提一句，但別過度延伸。

## 消費習慣記憶（habits.local.json，選填但推薦）

CardRadar 會「記住」你怎麼消費，好在每次回饋資料更新時，率先針對你的真實習慣做摘要與推薦。習慣資料存在**同目錄**的 `habits.local.json`（已被 `.gitignore` 排除、不上傳；範本＝ `habits.example.json`）。

三類習慣，每筆記 `count`（出現次數＝權重）與 `lastSeen`（最近一次日期）：
- `常用通路` — 你習慣消費的場域（全聯、超商、餐飲店、網購平台、加油站…）。
- `常用支付` — 你習慣的付款方式（實體刷卡，或 `profile.paymentApps` 內的某個行動支付）。**未列在 `profile.paymentApps` 的支付不記**（見鐵則 2）。
- `消費面向` — 你的消費落在哪些「通路情境」分類（一般消費／指定通路／網購／行動支付／海外／超商量販／餐飲／醫療／交通／加油）。

選填：`月度概況`（各面向大略月消費，用於估實拿與封頂）、`偏好`（現金優先／累哩程／某點數體系，用於推薦加權）。

### 怎麼學習（每次互動後輕量更新）
- 每次 `/card <消費情境>` 查詢後：先 `user_time_v0` 取今天，把命中的「通路／面向／最後採用的支付」各自 `count +1`、`lastSeen` 設今天，寫回 `habits.local.json`。新出現的通路/面向就新增一筆。
- 每次 `/card 新增/更新` 寫回饋資料後：把該卡涉及的通路/面向也各記一筆（代表你關注這塊）。
- **累加、不覆蓋**：寫回前先讀現值再加，別把既有 `count` 洗掉。這份檔是「學習軌跡」，會頻繁被覆寫。
- 若 `habits.local.json` 不存在：第一次要記時，用 `habits.example.json` 為骨架建立它——**只寫真的發生的習慣，`_說明`／`範例-…` 鍵不要照抄**。
- 使用者可隨時手動編輯或修正這份檔；學習只做加總，不刪使用者手填的內容。
- 記習慣是「事後輕量副作用」，別因為要寫這檔而拖慢或打斷查詢回答；先回答，再靜默更新。

## 消費習慣摘要報告與推薦（率先對習慣做）

觸發時機（兩者都算）：
1. **`/card 摘要`（或 `/card 報告`）** — 使用者主動要一份總覽。
2. **每次更新回饋資訊之後自動做** — `/card 新增`／`/card 更新`／`/card 失效` 寫入並回讀確認後，**率先**針對受影響的消費習慣，自動附一段摘要報告與推薦，不必等使用者再問。

### 做法
1. 讀 `habits.local.json`（缺檔或全空 → 告知「還沒有習慣紀錄，先用幾次 `/card` 或手動填 `habits.local.json`」，不要硬編）。
2. 依 `count`（權重）由高到低排序習慣；近期 `lastSeen` 可再加權，久未出現的往後放。取權重前幾名的通路/面向為「重點習慣」。
3. 對**每個重點習慣**，照「查詢流程」讀即時 Notion 資料、套三鐵則，算出目前最優卡與實拿：有 `月度概況` 就用該金額估實拿與封頂點；沒有就給率＋上限＋封頂點。
4. **若是「更新回饋資訊」觸發**：先標出變化——哪個習慣的最優卡因這次更新換了、實拿變多/變少、或新加碼剛好命中你常用通路。把「因為這次更新，你某習慣現在該改刷某卡」放最前面；沒有任何習慣受影響就直說「這次更新不影響你目前的常用習慣」。
5. 套 `偏好`：實拿相近時依偏好（現金優先／累哩程…）決定推薦順序，並可點出點數折現難易（參 `profile.pointMappings`）。
6. 套鐵則 3：命中習慣的加碼若即將到期/已過期，於摘要裡提醒重查。

### 輸出格式（摘要）
先講「這次更新對你習慣的影響」（更新觸發時才有），再給「你的習慣 × 最優卡」清單（權重高者先），最後給行動建議。手機閱讀、簡潔，別把整張規則表倒出來。

```
更新影響：<這次改了什麼 → 對你哪個常用習慣、最優卡是否改變>（僅更新觸發時）
你的習慣最優卡：
· <常用通路/面向1（權重高者先）>：<最優卡>，實拿約 <金額>（<率>×<月量>，<上限狀態>）
· <常用通路/面向2>：<最優卡>，實拿約 <金額>
建議：<最該調整的一兩個動作，例如某通路改用某卡、某加碼即將到期要重查>
注意：<到期／支付限制 等提醒>
```

## 查詢流程
1. 讀 `config.local.json` 與 `profile.local.json`（缺檔則依上方提示停止）；有 `habits.local.json` 一併讀入備用。
2. `user_time_v0` 取今天（判到期用）。
3. 讀 Notion：`notion-search`（帶 `collection://<config.ruleDataSourceId>`）找相關列 → `notion-fetch` 逐列讀；
   或用 `config.rowIndex` 的 page ID 直接 fetch。不要預設用整表 SQL（部分帳號被擋）。排除「已失效」。
   先比對「通路特例」——若情境命中全聯這類特例，直接套已知現實，別只看帳面率。
4. 套鐵則 2：剔除依賴 `profile.paymentApps` 以外支付的加碼列。
5. 套鐵則 1：用使用者給的金額算每張候選的實拿；沒給金額就給率＋上限＋封頂點。
6. 依實拿排名。
7. 套鐵則 3：標到期/過期提醒，已失效不列入。
8. 回答後：把命中的通路/面向/採用的支付更新進 `habits.local.json`（見「消費習慣記憶 → 怎麼學習」），靜默進行、別拖慢回答。

## 輸出格式
先講結論（贏家＋實拿＋為什麼），再講次選，最後才是注意事項。手機閱讀，簡潔。

```
這筆用 <卡片> 最優：實拿約 <金額>（<率> × <消費>，<上限狀態>）。
次選 <卡片>：實拿約 <金額>。
注意：<到期／支付限制 等提醒>
```

別長篇大論、別把整張表倒給他。只回他問的情境，給可行動的答案。

## 寫入操作（新增／更新／失效）

寫入沿用三鐵則的精神。最重要的原則：**更改或新增信用卡資訊時，有疑義者務必主動提出並當場向使用者釐清，問清楚再寫，絕不先猜一個草稿放著。**
常見要釐清的疑點：卡別／版本（現金版/點數版、新戶/舊戶）、會員等級與戶況、官方條件彼此衝突或寫得模糊、通路該歸哪一類、加碼是否依賴使用者沒在用的支付。
有疑惑就先問（通常一兩題就夠），問清楚再寫；無疑惑就直接寫。
所有新增或實質更新的列：狀態＝「生效」、`最後驗證日`＝今天（記錄這列最後一次對官方查證的日期）、`來源連結`放官方頁。

**寫入並回讀確認後，率先對消費習慣做摘要**：把這次涉及的通路/面向記進 `habits.local.json`，接著照「消費習慣摘要報告與推薦」自動附一段——標出這次更新對使用者常用習慣的影響（最優卡是否換、實拿變化），沒受影響就直說不影響。這一步是「更新回饋資訊」的預設收尾，不必等使用者再問。

### DB schema（寫入時的欄位與型別）
一列＝一張卡的一條通路規則。屬性名稱要逐字相符：

| 屬性 | 型別 | 值 |
|---|---|---|
| 規則名稱 | title | 如「大戶卡-國內一般消費」 |
| 卡片 | select | 既有卡之一；新卡見下方註 |
| 銀行 | select | 依你的卡所屬銀行 |
| 通路情境 | select | 一般消費／指定通路／網購／行動支付／海外／超商量販／餐飲／醫療／交通／加油 |
| 回饋率(%) | number | 純數字，如 3.5 |
| 回饋形式 | select | 現金／LINE POINTS／小樹點／OpenPoint／樂天點數／哩程／其他 |
| 每月回饋上限(元) | number | 空＝無上限 |
| 條件 | multi-select | 新戶／綁定／登錄／自動扣繳／最低門檻／穿戴裝置／指定行動支付／無上限 |
| 排除項目 | text | 細則、門檻、適用對象、⚠註記都寫這 |
| 活動到期日 | date | 加碼到期；長期回饋放遠日如 2026-12-31 |
| 來源連結 | url | 官方頁優先 |
| 最後驗證日 | date | 寫入/更新當天（最後一次對官方查證的日期）|
| 狀態 | select | 生效／即將到期／已失效 |

### 寫入機制（這些坑要記住）
- **新增列**：`notion-create-pages`，parent＝`{"type":"data_source_id","data_source_id":"<config.ruleDataSourceId>"}`，properties 用上表欄位。
- **日期欄**要拆兩鍵：`"date:活動到期日:start":"YYYY-MM-DD"` ＋ `"date:活動到期日:is_datetime":0`；`最後驗證日` 同理。
- **條件（multi-select）**用 JSON 陣列字串：`"條件":"[\"登錄\",\"最低門檻\"]"`。
- **multi-select 新選項無法在寫入時自動建**（條件那欄若要用沒列過的標籤，先請使用者在 Notion UI 加選項）。
- **新卡的「卡片」select 選項**：先試寫；若該卡選項不存在導致寫入失敗，請使用者在 Notion UI 的「卡片」屬性新增該卡名，再重寫。
- **更新單列**：`notion-update-page`，`command:"update_properties"`，帶 `page_id` 與要改的 properties。
- **更新 text 欄（排除項目）會整欄覆蓋**：先 `notion-fetch` 讀現值，把舊內容＋新註記一起寫回，別只寫新的。
- 改頁面內文/表格 cell 用 `replace_content`（`update_content` 對 table cell 會靜默失敗）；但本表是 DB 屬性，用 `update_properties` 即可。
- **寫入後一定 `notion-fetch` 回讀確認**，再回報使用者。

### 流程：`/card 新增 <卡名>`
1. 先確認卡別與版本（現金版/點數版、會員等級、新戶/舊戶）。**有任何疑惑當場問清楚再往下，別猜。**
2. web 查官方公告（銀行官網優先，比價站次之），抓各通路的率/上限/條件/排除/到期日。
3. 一張卡通常拆**多列**（一般消費、指定通路加碼、海外、特定通路…各一列）。
4. 若 web 查到的條件彼此衝突、寫得模糊或對不上使用者的卡別，**先回頭問使用者釐清，不要硬寫**。
5. `notion-create-pages` 寫入：狀態「生效」、最後驗證日＝今天、來源放官方。
6. 套 `profile` 前提（會員等級、幣倍/Level 上限、實際支付、電帳+自扣成立…）標進排除欄；他沒用的支付加碼要註明無效。
7. 回讀確認後回報各列摘要。

### 流程：`/card 更新 <卡名/通路>`
1. `notion-search`／`config.rowIndex` 找到該卡的列，`notion-fetch` 讀現值。
2. web 重查官方最新條件。
3. 若新舊條件對不上、官方寫得模糊或不確定要改哪一列，**當場問使用者釐清再改**。
4. `update_properties` 改動欄（率/上限/條件/排除/到期日），`最後驗證日`＝今天，狀態維持「生效」。
5. 活動已結束 → 狀態改「已失效」，或直接改寫成新一檔的內容（一樣維持「生效」）。
6. 回讀確認、回報前後差異。

### 流程：`/card 失效 <列>`
- 活動結束，狀態→「已失效」（查詢時自動排除，不刪列以留歷史）。

## 不做的事
- 不憑記憶報數字（一定讀 Notion）。
- 不寫死 ID 或個人資料（一定讀 `*.local.json`）。
- 不把使用者沒用的支付的加碼算進去。
- 不只看比率排名（要算實拿）。
- 有疑惑不硬猜著寫——當場問使用者釐清再寫。
- 新增/更新前不憑印象填數字——一定先 web 查官方再寫。
- 更新 text 欄（排除項目）前不先讀現值就覆蓋（會洗掉舊註記）。
- 寫入後不略過回讀確認。
- 更新回饋資訊後不忘率先對消費習慣做摘要與推薦。
- 學習習慣時不覆蓋洗掉既有 `count`／使用者手填內容（一律先讀再累加）。
- 不把使用者沒在用的支付記進 `常用支付`（比照鐵則 2）。
